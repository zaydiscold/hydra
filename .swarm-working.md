## NEW FINDINGS — Cross-Platform + DX (NF3)

### 🔴 CRITICAL (1)

75. **Windows: `child.kill('SIGTERM')` fails — shutdown leaks child processes** — `shutdown.js:16`. `killTrackedChildren()` uses `child.kill('SIGTERM')` but Windows has no SIGTERM signal. On Windows, `child.kill()` with a signal name throws or silently fails. Spawned `prisma db push` children (via `schemaSync.js:194`) are never killed on Windows, leaking processes. Should use `child.kill()` with no arguments (defaults to SIGTERM on POSIX, `TerminateProcess` on Windows) or handle per-platform.

### 🟠 HIGH (3)

76. **No hot reload in Electron dev — manual Ctrl+R required** — `package.json:34`. `dev:electron` uses `concurrently -k "vite" "electron ."` — when Vite HMR rebuilds the frontend, the Electron BrowserWindow does NOT auto-reload. Developers must manually press Ctrl+R in the Electron window after every frontend change. Missing `electron-reload`, `vite-plugin-electron-renderer`, or a custom watcher that calls `webContents.reload()` when Vite's `dist/` changes.

77. **Renderer never checks `result.ok` from preload bridge — silent failures** — `App.jsx:470,478`, `Diagnostics.jsx:34-36`, `Settings.jsx:43-45`. The preload bridge returns `{ok, data/error}` Result types, but ALL renderer call sites only wrap calls in `try/catch`. If the IPC handler returns `{ok: false, error: '...'}`, the `try/catch` succeeds and the error is silently swallowed. `quitApp()` failures, `appVersion()` failures, and diagnostic data failures are invisible.

78. **`env.js:54` backslash-to-forward-slash regex misses UNC paths** — `env.js:54`. `rawPath.replace(/\\/g, '/')` converts single backslashes but doesn't handle Windows UNC paths (`\\?\C:\Users\...`) which would produce `//?/C:/Users/...` — not a valid file: URL path. UNC paths can arise from `app.getPath('userData')` in certain Windows configurations (e.g., redirected folders, roaming profiles).

### 🟡 MEDIUM (7)

79. **`launch.js:143` double-escaped `\n` prints literal `\n` text** — `scripts/launch.js:143`. Uses `'Migration errors:\\n  '` (literal `\n` string) instead of `'Migration errors:\n  '` (actual newline). Migration error details show literal `\n` in console output instead of proper line breaks: `Migration errors:\n  - problem a\n  - problem b`.

80. **`openBrowser()` doesn't handle Linux — no `xdg-open` fallback** — `scripts/launch.js:80`. Uses `open` for non-Windows (macOS) with no Linux detection. On Linux, `open` may not exist or is an unrelated command. Should use `xdg-open` on Linux. Also, `start ""` on Windows is fragile with URLs containing `&` or special chars — should use `cmd.exe /c start ""`.

81. **`prepare-electron-resources.mjs` no fallback if LOCALAPPDATA unset on Windows** — `scripts/prepare-electron-resources.mjs:42-44`. Uses `process.env.LOCALAPPDATA` for Playwright cache on Windows but has no fallback if unset (e.g., on Server Core or minimal installs). Should fall back to `process.env.APPDATA` + `/../Local` or `homedir()/AppData/Local`.

82. **`dev:electron` doesn't set `VITE_DEV_SERVER_URL` — fails in certain setups** — `electron/main.js:164`. Falls back to `http://localhost:5173` hardcoded, but `vite.config.js` uses `HYDRA_VITE_PORT` (default 5173). If only one dev changes the Vite port via env var, Electron loads the wrong URL. `dev:electron` should pass through `HYDRA_VITE_PORT` or set `VITE_DEV_SERVER_URL` explicitly.

83. **`npm run test` API integration test always silently passes** — `server/tests/electron-api-integration.test.mjs:22-25`. Every test catches `fetch` errors and calls `assert.ok(true)`, so the test suite passes even when the server is completely down. This is a test anti-pattern — tests should be skipped explicitly (via `test.skip`) or the server check should happen in a `beforeEach` hook, not inside test assertions.

84. **No sourcemap config — production crashes are hard to debug** — `vite.config.js`. No `build.sourcemap` option set. Vite defaults to `false` for production, which is fine for distribution size but means any production crash has unreadable minified stack traces. Consider `sourcemap: 'hidden'` in production builds — generates `.map` files for error tracking but doesn't add `sourceMappingURL` comments to bundles (no exposure to clients).

85. **`hydra logs` reads entire file into memory** — `bin/hydra.mjs:112-116`. Uses `readFileSync(logPath, 'utf-8')` to load the entire log file then `.split('\n')` to get last 50 lines. If the log file grows to 500MB+, this OOMs the CLI process. Should use file-tail approach (seeking to end of file and reading backwards, or spawning `tail -n 50`).

### 🔵 LOW (8)

86. **`hydra` CLI only works via `npm link` with no setup automation** — `bin/hydra.mjs:71-73`. Help text says "Install once from the repo root: npm link". No `npm run link` or `postinstall` script to detect and suggest linking. Users who clone the repo won't discover this without reading the help output first.

87. **`hydra` default command starts prod server with no confirmation** — `bin/hydra.mjs:127-129`. Running `hydra` with no subcommand immediately starts a production server (`launch.js`). Most CLI tools show help on bare invocation. Beginners may be surprised when it starts a server and blocks their terminal.

88. **`hydra doctor` mislabels dir size as "Disk Info"** — `bin/hydra.mjs:83-88`. Shows `dataDir` size in MB labeled "Disk Info", but actual free disk space on the volume would be far more useful for diagnostics (e.g., warning when < 500MB free). The `statSync.size` reports directory metadata size, not its recursive content size.

89. **npm scripts poorly organized — no categories, confusing aliases** — `package.json:9-39`. `dev:web` duplicates `dev`. `preview`/`preview:electron`/`preview:static`/`start:electron` all overlap. No `--help` script, no `description` fields for `npm run` discoverability. Scripts intermix dev, test, build, and preview without grouping. Common DX expectation: `dev`, `dev:electron`, `build`, `test`, `lint`, `preview`.

90. **Win/Linux missing arm64 targets in electron-builder** — `electron-builder.yml:83-95`. Windows only targets x64 (no arm64 for Snapdragon laptops). Linux only targets x64 AppImage (no arm64 for Raspberry Pi / Apple Silicon Linux VMs). No .deb or .rpm packages — only AppImage on Linux, which some enterprise users can't install.

91. **No Windows code signing config — SmartScreen warnings** — `electron-builder.yml:81-88`. No `certificateFile`, `certificatePassword`, or `publisherName` for Windows code signing. Users will get "Windows protected your PC" SmartScreen warnings on install, hurting trust and adoption.

92. **`isPidAlive()` always returns false on Windows** — `schemaSync.js:83-90`. Uses `process.kill(pid, 0)` which is POSIX-only. On Windows, `process.kill(pid, 0)` throws `EPERM`. The code catches this and returns `false`, so PID-staleness detection always reports "dead" on Windows. This means orphaned migration locks on Windows are only broken by TTL expiry (60s), not by PID liveness checks.

93. **No test coverage for: CLI, preload/IPC, schema sync, afterPack, cross-platform paths** — Test gaps: `bin/hydra.mjs` (0 tests), `electron/preload.js` + `electron/app/ipc.js` (0 tests for contract/handlers), `electron/app/schemaSync.js` (no unit tests for hash computation, lock acquisition, fallback logic), `electron/builders/afterPack.js` (only smoke-checked at package time). No cross-platform path tests (Windows backslash, UNC, XDG, APPDATA).

## NEW FINDINGS — Performance/Optimization (NF1)

### 🟠 HIGH (4)

94. **PrismaClient instantiated at module level — blocks all imports for Prisma engine dlopen** — `server/services/db.js:1-4`. `const prisma = new PrismaClient()` executes at module evaluation time. The Prisma query engine binary (18MB darwin-arm64 `.dylib.node`, 93MB total `.prisma/client/` including all platforms) is dlopen'd synchronously during the `import('../server/index.js')` in `main.js:157`. This blocks the Electron main process during the splash→main transition window, since imports are synchronous and the engine load happens before `bootstrap()` even begins. Measured import+construct: ~28ms warm, but cold-cache first-load (filesystem + dlopen) is likely 50-200ms depending on disk speed. The import chain is: `main.js` → `server/index.js` → `server/services/store.js` → `server/services/db.js` → `@prisma/client` → native `.node` engine. Moving PrismaClient to lazy initialization (factory function or `await import()` in bootstrap) would let the splash render without blocking on engine load.

95. **App .dmg is 612MB — 54% is Chromium for Playwright** — `release/mac-arm64/Hydra.app/Contents/Resources/chromium/chrome-mac-arm64/` = 330MB. The Playwright browser binary ships with the app for the bulk-auth browser automation feature (used only in BulkAuthWizard, a rarely-triggered workflow). Additionally, `playwright-core` (8MB) and `playwright` (3.5MB) add ~12MB of JS overhead. The Chromium binary could be lazy-downloaded on first bulk-auth use via Playwright's built-in `npx playwright install chromium`, removing 330MB (54%) from the shipping package. The `build/electron/chromium/` directory in `extraResources` would be removed, and a first-run check + install prompt added.

96. **No vite build optimization — main bundle is 266KB monolithic** — `vite.config.js` has zero `build` configuration: no `rollupOptions`, no `manualChunks`, no `terserOptions`, no `minify`/`target` overrides. The main bundle (`dist/assets/index-Bfnp1XDb.js`, 266KB) inlines: React 19.2.4 (full production build ~120KB), ReactDOM (~130KB), React Router 7.13.2 (~80KB), plus App component and 11 icon components imported at the top of `App.jsx`. Total JS across all chunks: 441KB. CSS: 52KB. Adding vendor chunk splitting (`manualChunks: { vendor: ['react', 'react-dom', 'react-router-dom'] }`) would reduce initial parse time by ~200KB and enable cross-update browser caching (vendors change rarely). The `build.rollupOptions` field is completely missing from vite.config.js.

97. **No startup timing instrumentation anywhere** — Zero `performance.mark()` or `performance.measure()` calls in `main.js`, `server/index.js`, or any Electron module. Impossible to measure real-world startup performance, track regressions, or identify per-user bottlenecks. Key measurement points needed: splash shown, `ensurePackagedRuntimeState` done, `shouldSyncSchema` done, Prisma import start/end, `bootstrap()` start/end, `loadURL` start/end, `ready-to-show`. Without instrumentation, all startup performance discussions are speculation.

### 🟡 MEDIUM (4)

98. **server/index.js eagerly imports ALL 13 route + 11 service modules at startup** — `server/index.js:13-32`. Every route (auth, accounts, keys, dashboard, codes, generator, pool, proxy, webhooks, system, debug) and service (pinger, retention, task-supervisor, legacy-storage, store, session-refresher, proxy-gate, rotation-manager) is statically imported. The `store.js` → `db.js` → `PrismaClient()` path is the heavyweight. But all other modules expand the import graph unnecessarily. Routes like generator, debug, pool, and services like rotation-manager aren't needed until their first API call. Lazy-loading non-critical modules would reduce cold-start module resolution time.

99. **Redundant logging frameworks: electron-log + winston both in production** — `package.json:46,52`. The app ships two logging frameworks: `electron-log` (160KB in packaged app) for Electron main process logging + file transport, and `winston` (308KB in packaged app) for server-side logging. Both provide file transport, formatting, and level filtering. The server uses winston via `server/services/logger.js`, but the main process already has electron-log with file rotation configured at `env.js:17-20`. Consolidating on electron-log (which supports Node.js contexts via its `node` import) would remove the winston dependency and simplify the logging stack.

100. **Splash window runs 48 CSS-animated particles continuously during CPU-heavy startup** — `windows.js:46-48`. The splash HTML includes 48 `<span>` elements across 4 CSS animation types (rain, drift, spiral, pulse) all running `animation: ... infinite` simultaneously. During the critical cold-start window (Prisma engine load + server bootstrap + React render), these animations compete for GPU compositing + CPU scheduling with the actual startup work. On lower-end machines, this compounds startup latency. A simpler splash (static or single animation) or reducing to 8-12 particles would reduce resource contention without being visually noticeable.

101. **electron-log Object.assign(console, ...) adds overhead to every log call** — `electron/app/env.js:21`. `setupLogging()` calls `Object.assign(console, log.functions)` which replaces native `console.log/warn/error/info` with electron-log's instrumented wrappers. Every `console.log()` now goes through `processMessage()` → format pipeline → file transport. During startup, the server emits dozens of log lines (bootstrap, proxy keys, pool reload) all going through this double-write path (console + file). Consider using `log.initialize({ spyRendererConsole: false })` to avoid the console instrumentation, or scoping the override to only capture explicit log calls.

### 🔵 LOW (3)

102. **afterPack prunes .prisma/client/ engine binaries but @prisma/client/ still ships ~3.1MB of unused JS** — `electron/builders/afterPack.js:85-97`. The afterPack script correctly filters foreign engine binaries from `.prisma/client/` (keeping only darwin-arm64 .dylib.node), drops wasm variants, and prunes `@prisma/client/runtime/` of edge/browser files. However, `@prisma/client/` (3.1MB in packaged app) retains: generator scripts, TypeScript declaration `.d.ts` files, ESM/CJS dual builds (`.mjs`/`.js`), bundler-specific entrypoints (`edge.js`, `wasm.js`, `browser.js`), and runtime utilities not used in the Node.js sqlite context. Additional post-build filtering for unused `@prisma/client` files could save ~1-1.5MB.

103. **1.7MB image in dist/generated/ may be unused** — `dist/generated/hydra-icon-ai-2026-05-05.png` = 1.7MB. `dist/index.html` (705 bytes) doesn't reference it. If only used as app icon source (converted to .icns/.ico by electron-builder) it's dead weight in dist/. Verify usage and remove if unreferenced.

104. **electron-log file transport writes at 'info' level — floods startup I/O** — `electron/app/env.js:18`. `log.transports.file.level = 'info'` means every `console.log()` during startup triggers file I/O through electron-log's transport pipeline at the most I/O-intensive moment (server bootstrap, proxy key derivation, pool reload). Setting to `'warn'` in production would eliminate ~90% of startup writes without losing visibility into errors. Info-level logs could still go to console (ephemeral) without disk persistence.
